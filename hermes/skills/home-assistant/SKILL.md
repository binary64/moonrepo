# Home Assistant Control

## When to use

Any smart-home request — TV, lights, thermostats, sensors, scenes, scripts, automations, media playback. If the user says "turn on X", "play Y on the TV", "what's the temperature in Z", or "run the <scene> scene", load this skill.

## Connectivity

Hermes is a cluster-adjacent node, so HA is reachable at a ClusterIP with no auth proxy in the way:

```
HASS_URL=http://home-assistant.home-assistant.svc.cluster.local:8123
HASS_TOKEN=<long-lived access token>
```

Both come from `/home/hermes/.hermes/.env` (generated by `hermes/bootstrap-env.sh` from AWS Secrets Manager — never committed). Load them before running:

```bash
set -a; source ~/.hermes/.env; set +a
```

If either is missing, tell the user to run `~/moonrepo/hermes/bootstrap-env.sh` (which requires `./infra/secrets/set-secret.sh hermes-ha-token "<token>"` to have been run first).

## Cheat sheet

All examples assume `$HASS_URL` and `$HASS_TOKEN` are exported.

### List entities

```bash
curl -s -H "Authorization: Bearer $HASS_TOKEN" \
  "$HASS_URL/api/states" | jq -r '.[] | .entity_id' | sort
```

Filter to a domain:

```bash
curl -s -H "Authorization: Bearer $HASS_TOKEN" \
  "$HASS_URL/api/states" | jq -r '.[] | select(.entity_id | startswith("media_player.")) | .entity_id'
```

### Get a single entity state

```bash
curl -s -H "Authorization: Bearer $HASS_TOKEN" \
  "$HASS_URL/api/states/media_player.living_room_tv" | jq
```

### Call a service (the common workhorse)

```bash
curl -s -H "Authorization: Bearer $HASS_TOKEN" \
     -H "Content-Type: application/json" \
     -d '{"entity_id":"media_player.living_room_tv"}' \
     "$HASS_URL/api/services/media_player/turn_on" | jq
```

Domain/service pairs you'll use a lot:

| Action | Domain | Service | Extra data |
|--------|--------|---------|------------|
| TV on  | `media_player` | `turn_on` | — |
| TV off | `media_player` | `turn_off` | — |
| Volume | `media_player` | `volume_set` | `{"volume_level": 0.4}` |
| Play source | `media_player` | `select_source` | `{"source": "Netflix"}` |
| Play media | `media_player` | `play_media` | `{"media_content_id":"...","media_content_type":"..."}` |
| Light on | `light` | `turn_on` | `{"brightness_pct": 60, "color_name": "warm_white"}` |
| Scene | `scene` | `turn_on` | entity_id is the scene |
| Run script | `script` | `turn_on` | entity_id is the script |
| Toggle switch | `switch` | `toggle` | — |

### Discover what a device supports

Before acting blind, read its state — the `attributes` dict lists `supported_features`, `source_list`, etc.

```bash
curl -s -H "Authorization: Bearer $HASS_TOKEN" \
  "$HASS_URL/api/states/media_player.living_room_tv" | jq '.attributes'
```

### Templating / logbook / history

- `/api/template` — render a Jinja template server-side
- `/api/history/period` — historical state
- `/api/logbook` — human-readable log

## Workflow

1. Load env: `set -a; source ~/.hermes/.env; set +a`
2. If the user's target isn't explicit, list candidate entities first and confirm.
3. Make the service call.
4. Read back the state to verify the change actually happened (HA calls are async — `turn_on` returns 200 before the device responds).

## Pitfalls

- **200 ≠ worked.** Always verify with `/api/states/<entity>` after a service call.
- **TV wake via HDMI-CEC or WoL** may take 3–8s; poll state with a small sleep loop.
- **Long-lived tokens don't expire** but are tied to a user — if someone revokes the HA user, `bootstrap-env.sh` needs to re-pull a new token.
- **`select_source` names are case/space-sensitive** — grab them from `.attributes.source_list`.
- **Don't log the token.** When echoing curl commands for the user, use `$HASS_TOKEN` literal, not the expanded value.

## Rotating the token

```bash
# In HA UI: Profile → Security → create new long-lived access token
cd ~/moonrepo/infra/secrets
./set-secret.sh hermes-ha-token "<new-token>"
~/moonrepo/hermes/bootstrap-env.sh
# optionally revoke the old token in HA UI
```
